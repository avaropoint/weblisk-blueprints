# Project Structure

> How to organize a blueprint-driven Weblisk project.

## Directory Layout

```
my-project/
  .weblisk/
    config.yaml             # Hub configuration (ports, LLM provider, env)
  blueprints/
    global.yaml             # Project identity, brand, policies
    code.yaml               # Code generation conventions
    theme.yaml              # Design tokens (derived from global brand)
    pages/                  # One file per route
      home.yaml
      about.yaml
      blog.yaml
    components/             # Reusable UI elements
      nav.yaml
      footer.yaml
      card.yaml
    islands/                # Interactive regions (agent-bound)
      contact-form.yaml
      live-feed.yaml
    content/                # Structured content and copy
      site-copy.yaml
      features.yaml
    connections/            # External service integrations
      analytics.yaml
    assets/                 # Asset declarations and generation rules
      brand.yaml
  assets/                   # Physical files (images, fonts, video)
    logo.svg
    hero.webp
    fonts/
      inter-var.woff2
  domains/                  # Server-side domain controllers (if needed)
    content/
      domain.yaml
  agents/                   # Server-side agents (if needed)
    chat-responder/
      agent.yaml
  public/                   # Generated output (build artifact)
    index.html
    css/
    js/
    assets/
```

## Key Conventions

### `blueprints/` Is Source Code

Everything in `blueprints/` is human-authored YAML. This is what
developers edit. It IS the project specification.

### `public/` Is Build Output

Everything in `public/` is generated by the LLM from blueprints. It can
be deleted and regenerated at any time. It should be in `.gitignore`
unless you want to version the output.

### `assets/` (Root) Is Physical Media

Files that can't be generated — photographs, licensed fonts, brand
logos, video. These are referenced by blueprints but not produced by
the generation pipeline.

### `blueprints/assets/` Is Asset Declarations

Describes how physical assets are used, what sizes to generate for
responsive images, what favicon set to produce, etc.

### `.weblisk/config.yaml` Is Runtime Config

Not a blueprint — this is operational configuration. Ports, LLM
provider API keys, environment-specific settings. Exists only if the
project has server-side components.

## File Naming

| Type | Convention | Example |
|------|-----------|---------|
| Pages | Match route slug | `pricing.yaml` → `/pricing` |
| Components | Describe the element | `nav.yaml`, `card.yaml` |
| Islands | Describe the interaction | `live-feed.yaml`, `quote-calculator.yaml` |
| Content | Describe the collection | `features.yaml`, `team.yaml` |
| Connections | Name the service | `stripe.yaml`, `analytics.yaml` |

## Minimal Projects

### Client-Only (Static Site)

```
my-site/
  blueprints/
    global.yaml
    code.yaml
    pages/
      home.yaml
  public/                   # generated
```

Three files. That's a complete project. The LLM generates a full static
page from `global.yaml` (brand, identity) + `code.yaml` (conventions) +
`home.yaml` (content and layout).

### Client + Island

```
my-site/
  blueprints/
    global.yaml
    code.yaml
    pages/
      home.yaml
    islands/
      chat.yaml
  .weblisk/config.yaml      # needed for agent connection
  agents/
    chat-responder/
      agent.yaml
  public/                   # generated
```

Adds one interactive region. The island connects to a server-side agent.
The `.weblisk/config.yaml` declares the hub for the agent to register with.

## Blueprint Resolution Order

When generating output, the pipeline loads blueprints in this order:

1. `global.yaml` — Always loaded first (project-wide context)
2. `code.yaml` — Always loaded second (constrains all generation)
3. `theme.yaml` — If present (design token context)
4. Target blueprint — The specific file being generated
5. Referenced blueprints — Anything the target imports/extends

This order ensures every generated file inherits project-wide rules
without the developer repeating them.
