# Template: Hello World

The simplest possible Weblisk application. A single HTML page served
through the application gateway, with one domain controller and one
work agent demonstrating the complete request flow: browser → gateway →
domain → agent → island render.

This is the starting point for every Weblisk application. Everything
on the marketplace — every template, every starter — extends this
foundation.

---

## Overview

This template produces a complete Weblisk hub that serves a single
web page. The page contains one island that fetches a greeting from
the server and renders it in the browser. Everything runs locally —
orchestrator, gateway, one domain controller, and one work agent.

### What It Demonstrates

- The Weblisk server lifecycle (orchestrator → agent registration → ready)
- The application gateway serving static HTML and routing island requests
- The Weblisk client framework discovering and hydrating islands
- A domain controller dispatching work to an agent
- An agent executing a task and returning a result
- The complete round-trip from browser to agent and back

### What It Does NOT Include

- Authentication (no login — all routes are public)
- Multiple pages or navigation
- Multiple domains or agents
- Infrastructure agents (cron, alerting, etc.)
- Federation or multi-hub connectivity
- Persistence (all state is in-memory)

## Gateway Configuration

### Route Table

```yaml
routes:
  # Landing page
  - path: /
    target: static:filesystem
    file: index.html
    auth: none

  # Static assets (CSS, JS, images)
  - path: /assets/*
    target: static:filesystem
    auth: none
    cache: immutable

  # Island data routes
  - path: /islands/:page/:island
    target: resolve:island
    auth: none                     # Public for hello-world
```

### Island Routing

```yaml
islands:
  home:
    greeting:
      target: domain:greeting
      action: hello
```

When the browser requests `GET /islands/home/greeting`, the gateway:

1. Matches the `/islands/:page/:island` route
2. Resolves `home:greeting` from the island routing table
3. Finds target `domain:greeting`, action `hello`
4. Builds a TaskRequest and routes it to the greeting domain controller
5. Returns the domain's response as JSON to the browser

## Static Page

### public/index.html

```html
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="UTF-8">
  <meta name="viewport" content="width=device-width, initial-scale=1.0">
  <title>Hello World — Weblisk</title>
  <link rel="stylesheet" href="/assets/styles.css">
</head>
<body>
  <main>
    <h1>Hello, Weblisk</h1>
    <p>Your application is running.</p>

    <!-- Island: greeting -->
    <!-- The Weblisk client framework will:                     -->
    <!--   1. Find this element by its data-island attribute    -->
    <!--   2. Fetch JSON from the data-src URL                  -->
    <!--   3. Render the response into this element             -->
    <section data-island="greeting"
             data-src="/islands/home/greeting">
      <p class="loading">Loading...</p>
    </section>
  </main>

  <footer>
    <p>Powered by <a href="https://weblisk.dev">Weblisk</a></p>
  </footer>

  <!-- Weblisk client framework — zero dependencies -->
  <!-- Source: https://github.com/avaropoint/weblisk -->
  <script src="/assets/weblisk.js"></script>
</body>
</html>
```

The `data-island` attribute marks an element as an island. The
`data-src` attribute tells the client framework where to fetch data.
The framework discovers all islands on page load, fetches their data
concurrently, and renders the responses into the DOM. The "Loading..."
fallback is shown until the island's data arrives.

## Domain: Greeting

A minimal domain controller with one workflow and one required agent.

### Configuration

```yaml
# domains/greeting/blueprint.yaml

domain:
  name: greeting
  version: 1.0.0
  description: Handles greeting requests

workflows:
  hello:
    description: Returns a greeting message
    phases:
      - name: greet
        agent: greeter
        action: generate-greeting

required_agents:
  - greeter
```

### Behavior

1. Receives a TaskRequest from the gateway with `action: "hello"`
2. Matches the `hello` workflow
3. Dispatches phase `greet` to the `greeter` agent via `POST /v1/execute`
4. Returns the agent's result to the gateway

### Registration

On startup, the greeting domain controller:

1. Generates an Ed25519 key pair (or loads from `.weblisk/keys/`)
2. Registers with the orchestrator at `POST /v1/register`
3. Reports status `degraded` until all required agents are online
4. Transitions to `online` when the `greeter` agent registers

## Agent: Greeter

A minimal work agent that returns a greeting message.

### Configuration

```yaml
# agents/greeter/blueprint.yaml

agent:
  name: greeter
  type: work
  domain: greeting
  version: 1.0.0
  description: Returns a greeting message with timestamp

capabilities:
  actions:
    - name: generate-greeting
      description: Returns a greeting message
      input: {}
      output:
        type: object
        fields:
          message:
            type: string
            description: The greeting text
          timestamp:
            type: string
            format: iso8601
            description: When the greeting was generated
          server:
            type: string
            description: Server version identifier
```

### Protocol Endpoints

The greeter agent implements the standard Weblisk agent protocol:

| Method | Path | Purpose |
|--------|------|---------|
| POST | `/v1/execute` | Handle task execution |
| POST | `/v1/health` | Return health status |
| POST | `/v1/describe` | Return agent manifest |

### Execute: generate-greeting

**Request** (from domain controller):

```json
{
  "task_id": "task-a1b2c3d4",
  "action": "generate-greeting",
  "payload": {}
}
```

**Response**:

```json
{
  "task_id": "task-a1b2c3d4",
  "status": "complete",
  "output": {
    "message": "Hello, World!",
    "timestamp": "2026-04-28T12:00:00Z",
    "server": "weblisk/1.0.0"
  }
}
```

### Health

**Response** to `POST /v1/health`:

```json
{
  "status": "ok",
  "component": "greeter",
  "version": "1.0.0",
  "uptime_seconds": 42
}
```

### Describe

**Response** to `POST /v1/describe`:

```json
{
  "name": "greeter",
  "type": "work",
  "version": "1.0.0",
  "domain": "greeting",
  "capabilities": {
    "actions": ["generate-greeting"]
  },
  "public_key": "<ed25519-public-key>"
}
```

### Registration

On startup, the greeter agent:

1. Generates an Ed25519 key pair (or loads from `.weblisk/keys/`)
2. Registers with the orchestrator at `POST /v1/register`
3. Begins accepting requests on its assigned port
4. The greeting domain detects the registration and transitions to `online`

## Data Flow

Complete step-by-step request flow for the greeting island:

```
 1. User navigates to http://localhost:8080/

 2. Gateway matches route "/" → static:filesystem → serves public/index.html

 3. Browser parses HTML, requests /assets/styles.css and /assets/weblisk.js
    Gateway serves both from public/assets/

 4. Weblisk client framework initializes:
    a. Scans DOM for [data-island] elements
    b. Finds <section data-island="greeting" data-src="/islands/home/greeting">
    c. Sends GET /islands/home/greeting to the gateway

 5. Gateway matches route "/islands/:page/:island":
    a. Resolves page=home, island=greeting from the island routing table
    b. Target: domain:greeting, action: hello
    c. Builds TaskRequest: { action: "hello", payload: {} }
    d. POST /v1/execute to the greeting domain controller

 6. Greeting domain controller:
    a. Matches workflow "hello"
    b. Phase 1: dispatch to greeter agent, action generate-greeting
    c. POST /v1/execute to the greeter agent

 7. Greeter agent:
    a. Receives the task request
    b. Generates { message: "Hello, World!", timestamp: "...", server: "weblisk/1.0.0" }
    c. Returns the result

 8. Domain controller returns the result to the gateway

 9. Gateway returns JSON response to the browser

10. Weblisk client framework:
    a. Receives the JSON response
    b. Renders the greeting data into the island's DOM region
    c. Replaces "Loading..." with the rendered greeting
```

## Running

### Using the CLI

```bash
# Create a new project from this template
mkdir hello-world && cd hello-world
weblisk server init --template hello-world --platform go

# Start everything (orchestrator, gateway, domain, agent)
```

Open `http://localhost:8080` in a browser.

## Next Steps

Once the hello-world is running, extend it:

- **Add authentication** — Apply `patterns/auth-session` to require
  login and protect island routes
- **Add a second page** — Create another HTML file with its own
  islands and add routes to the gateway
- **Add a real domain** — Replace the greeting domain with business
  logic (e.g., content management, analytics)
- **Add infrastructure agents** — Enable `cron` for scheduled tasks
  and `alerting` for notifications
- **Try a different platform** — Run the same template on Cloudflare
  Workers (`--platform cloudflare`) or Node.js (`--platform node`)
- **Deploy to production** — Apply `patterns/deployment` for TLS,
  process management, and monitoring

Each step builds on this foundation. The architecture scales from
this single-page hello-world to a full enterprise application with
multiple domains, dozens of agents, and federation across hubs.
